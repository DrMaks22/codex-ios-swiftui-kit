# Review Checklist

Use this order for final review:

1. state ownership
2. navigation/presentation lifecycle
3. view identity/composition
4. API availability
5. accessibility
6. performance
7. visual polish

## Severity and verdict

- use `../../../references/review-severity.md` mapping for each finding
- verdict required at end: Pass / Pass with fixes / Needs revision

## Findings format

- lead with highest severity
- include exact pattern or region
- include concrete fix
- keep residual risks explicit

## Patch order

1. compile blockers
2. flow blockers
3. accessibility blockers
4. stability/perf blockers
5. polish

## Guardrail

- do not invent findings when evidence is weak
